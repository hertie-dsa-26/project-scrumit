How Our Model Works

A simple explanation of how we built an AI system to detect money laundering in any given transaction.

The Challenge: Banks process millions of transactions daily, but catching money laundering is like finding a needle in a haystack. Our model learns from historical patterns to automatically flag suspicious transactions before they slip through the system.

How we built it: step by step

1

Data Cleaning

We started with more than a million synthetic transactions from the IBM AML dataset. We cleaned up missing values, standardized dates and amounts, and added geographic information. This ensures our model trains on high-quality data.

2

Feature Engineering

Raw transaction data isn't enough. We created features from the data that highlight suspicious patterns:

  • Payment Format: Which payment method was used? (Wire, ACH, Credit Card, Cash, etc.)
  • Currency: What currencies were involved? Some are linked to higher risk.
  • Transaction Amount: Is the amount unusual or typical?
  • Geographic Patterns: Where are the sending and receiving banks located?
3

Handling Imbalance

In real-world data, only ~0.1% of transactions are actual money laundering. This imbalance means a naive model could just say "not laundering" for everything and be over 99% right! We fixed this using SMOTE (Synthetic Minority Over-sampling), which artificially creates more examples of money laundering to help the model learn better.

4

Model Selection: Quadratic Discriminant Analysis (QDA)

We chose QDA after benchmarking several algorithms. QDA is a powerful yet interpretable model that assumes each class (laundering vs non-laundering) has its own Gaussian distribution. This allows it to capture complex patterns without overfitting:

  • Fast: It doesn't need extensive tuning or massive compute power.
  • Accurate: For this data, it outperforms more complex algorithms.
  • Probabilistic: We get a confidence score (0-100%) for each prediction, not just yes/no.
5

Tuning & Validation

We tested the model with 5-fold cross-validation to make sure it performs consistently. We also tuned the regularization parameter to prevent overfitting — this helps the model generalize to new, unseen transactions.

6

Feature Importance Analysis

Not all features matter equally. Using permutation importance, we identified which features the model relies on most. This helps compliance teams understand what patterns signal money laundering.

How Compliance Teams Use This

In the FRISK dashboard, compliance analysts:

  • Input transaction details (amount, banks, format, currency)
  • Get back a risk score from 0% to 100%
  • See which features triggered the alert
  • Review historical patterns to make informed decisions

What This Model Can't Do

  • Catch everything: Even 92% accuracy means 8% of money laundering slips through (false negatives).
  • Predict the future: Criminals adapt. The model learns from past patterns, not future schemes.
  • Replace human judgment: This is a tool to augment compliance teams, not replace them.
